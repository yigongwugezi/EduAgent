# EduAgent 第二阶段任务说明：数据真实化与核心智能体闭环

更新时间：2026-06-16 14:34  
适用对象：组长、前端成员、后端成员  
阶段定位：第一阶段已经跑通基础闭环，第二阶段要把系统从“页面能展示”升级为“数据真实更新、核心智能体可持续工作”。

## 1. 第二阶段为什么要做

第一阶段主要解决：

- 用户输入学习情况。
- 系统生成学生画像。
- 系统生成学习路径。
- 系统生成资源库。
- 前端页面能展示结果。

第二阶段要解决：

- 用户不是只说一次，后面会不断补充和纠正。
- 学生画像不能只生成一次，而要随对话持续更新。
- 学习路径不能永远固定，而要根据新画像、新反馈动态调整。
- 资源库不能只展示第一次生成的内容，而要根据后续需求持续更新。
- 页面上展示的数据要能证明是真的更新，而不是固定 mock 或壳子数据。
- 智能体不要先堆数量，而是先把 2-3 个核心智能体做实。

所以第二阶段的核心不是“再做几个页面”，而是：

> 先把数据本身做实，再围绕真实数据建立核心智能体闭环，让画像、路径、资源和学习反馈能围绕同一个 session 持续更新。

## 1.1 第二阶段优先级调整

结合组员讨论，第二阶段优先级调整为：

```text
数据真实化 > 核心智能体接口化 > 页面同步展示 > 更多智能体扩展
```

也就是说：

- 不是先堆很多智能体。
- 不是先做很多看起来丰富的页面。
- 不是继续依赖 mock 数据撑效果。

而是先证明：

- 当前 session 里确实保存了用户对话和生成结果。
- 画像确实根据后续对话更新。
- 学习路径确实基于当前画像生成。
- 资源库确实根据新需求变化。
- 页面展示的是当前 session 的最新数据。
- 后端接口能清楚触发和返回智能体结果。

## 2. 结合组员提出的问题

组员提到的几个问题，可以归纳成第二阶段的四条主线：

### 2.1 “这个智能体用一次之后好像不能更新了”

这说明第二阶段必须做 **持续画像更新机制**。

不是用户第一次说完就结束，而是每轮对话都要判断：

- 有没有新的画像信息？
- 有没有纠正旧信息？
- 有没有新的薄弱点？
- 有没有新的偏好？
- 是否需要重新生成路径或资源？

### 2.2 “资源库一次生成后，下次对话说其他的不会更新”

这说明第二阶段必须做 **资源库增量更新机制**。

例如：

- 用户先说想学数据结构。
- 系统生成基础讲义、题目、路径。
- 用户后面说“我链表不会，多给我题”。
- 资源库应该新增链表练习题，而不是还停留在第一次生成的资源。

### 2.3 “搭建数据库或者把其他智能体搭建起来？”

这两个都要做，而且在代码上不是完全分开的。

第二阶段优先做“真实数据链路 + 智能体接口化”。原因是：

- 如果没有持久化，就很难证明数据真的更新。
- 如果页面一直读 mock，就无法判断智能体是否真的生效。
- 如果 session 不隔离，不同用户或不同对话会互相污染。
- 如果数据库没有记录，刷新页面后就无法验证持续学习。
- 如果智能体没有通过稳定 API 暴露给前端，页面也很难验证智能体是否真的工作。

第二阶段应该分成：

- 后端负责智能体接口化、服务编排和基础持久化。
- 组长负责核心 agent 策略、测试集和协同设计。
- 前端负责交互分析、状态展示和页面更新。

智能体也要做，但第二阶段先做“少而实”的核心智能体，而不是一口气堆十个。

结合陶畅的意愿，后端成员可以负责“接口 + 智能体服务编排”这一块。因为在当前项目里，API 和智能体调用基本是一体的：前端请求接口，后端接口调用 AgentOrchestrator，智能体返回画像、路径和资源，再由接口返回给前端。

数据库不用被理解成“必须先做一个完整后台管理系统”。第二阶段可以先做基础版：

- session 持久化。
- 聊天记录持久化。
- 画像/路径/资源结果持久化。
- 后续数据多起来后再逐步加表和查询能力。

### 2.4 “允许 agent 之间相互调用可能挺有意思”

这个方向是对的，但第二阶段不要一下子做太复杂。

建议先实现轻量版：

```text
IntentAgent
  -> ProfileAgent
  -> KnowledgeAgent
  -> DiagnosisAgent
  -> PlannerAgent
  -> ResourceAgent
  -> ReviewAgent
```

每个 agent 不直接乱调别人，而是通过统一的 `AgentOrchestrator` 共享上下文。

第二阶段目标不是做一个完全自由的 agent 社会，而是做一个可控、可解释、可测试的多智能体工作流。

第二阶段核心智能体建议先固定为 3 类：

1. 画像更新智能体
   - 负责从多轮对话中抽取、补充、纠正学生画像。

2. 知识诊断智能体
   - 负责根据课程知识库和画像判断薄弱点。

3. 路径资源智能体
   - 负责根据画像和诊断结果生成学习路径与资源。

其余智能体可以保留接口和设计说明，但不作为第二阶段强制主线。

## 2.5 “导图和多模态资源先缓缓，因为 DS API 干不了这个”

这个问题也需要纳入第二阶段安排。

DeepSeek 当前更适合：

- 生成讲义文本。
- 生成练习题。
- 生成代码案例。
- 生成学习路径。
- 生成思维导图的 Mermaid/Markdown 结构。

但 DeepSeek 不适合直接生成真实视频或动画。多模态视频/动画应该后续接 SeeDance、讯飞多模态工具，或者先生成“视频脚本/分镜说明”。

因此第二阶段资源生成采用分层策略：

1. 文本类资源先做实
   - 课程讲义。
   - 练习题。
   - 拓展阅读。
   - 代码实操案例。

2. 结构化可视资源先做可渲染版本
   - 思维导图：智能体生成 Mermaid。
   - 知识图谱：智能体生成节点/边 JSON。
   - 前端负责渲染。

3. 多模态资源先保留接口和脚本
   - 第二阶段先生成视频脚本。
   - 第二阶段先生成分镜。
   - 页面标注“待接入多模态生成工具”。
   - 不假装已经生成真实视频。

## 2.6 “公开课文字不够，能不能让 AI 检索并加入视频上下文”

这个问题可以纳入知识库建设策略，但要分阶段做。

第二阶段不建议直接做“全网自动搜视频、下载视频、分析视频内容、自动入库”的完整链路，因为这会涉及：

- 视频平台版权和下载限制。
- 视频语音转文字。
- 视频内容质量判断。
- 视频切片和知识点对齐。
- 多模态模型调用成本。

但第二阶段可以先做“视频资源入库的基础版”：

1. 视频作为资源类型
   - 资源库里增加 `video` 或 `video_script` 类型。
   - 保存视频标题、链接、来源、适用知识点、推荐原因。
   - 不直接存大视频文件。

2. 视频上下文结构化
   - 如果能拿到字幕或简介，就让 AI 提取知识点。
   - 如果只有标题和简介，就先做弱结构化标注。
   - 字段包括：对应课程、章节、知识点、难度、适合人群、观看目的。

3. 教材与视频分层
   - 教材/讲义：作为权威知识来源。
   - 视频：作为辅助讲解资源。
   - AI 生成内容：作为个性化补充。

4. 后续扩展方向
   - 接入视频字幕解析。
   - 接入多模态模型分析视频片段。
   - 接入 SeeDance/讯飞多模态生成讲解视频。

因此第二阶段资源库可以先分成：

```text
教材/讲义资源
练习题资源
代码实验资源
思维导图资源
视频链接/视频脚本资源
```

这样既回应了“资源库要更丰富”的问题，又不会在第二阶段被视频处理拖死。

## 3. 第二阶段总目标

第二阶段目标：

> 实现真实数据驱动的多轮学习闭环：对话进入数据库，画像持续更新，路径和资源基于最新画像生成，页面展示当前 session 的最新结果。

第二阶段验收结果：

用户连续对话时，系统可以做到：

1. 记住之前说过的信息。
2. 正确更新新的画像字段。
3. 用户纠正时覆盖旧信息。
4. 根据新画像重新调整路径。
5. 根据新需求更新资源库。
6. 前端页面展示最新结果。
7. 数据能保存在本地数据库中，刷新页面不丢。
8. 页面能区分真实数据、智能体生成数据和 mock 兜底数据。

## 4. 三人任务量均衡分工

第二阶段不建议把任务分成“组长做智能体，别人随便做一点”。这样会导致任务量严重不平衡。

建议按 **三个等量模块** 分：

| 成员 | 负责方向 | 核心产出 |
|---|---|---|
| 组长 | 核心智能体策略与测试 | 画像更新策略、诊断/路径资源协同设计、测试集 |
| 后端 | 智能体接口化与数据链路 | AgentOrchestrator 接口、API、session 数据隔离、基础持久化 |
| 前端 | 交互闭环与数据可信展示 | 页面刷新、交互分析、数据来源展示、资源详情与演示体验 |

三个人都要做核心功能，不是只有组长做主线。

## 5. 组长任务：智能体协同与测试

### 5.1 要做什么

组长负责“核心智能体策略是否合理、测试是否覆盖真实使用场景”。

具体任务：

1. 升级画像抽取逻辑
   - 支持短句输入：`大一 学生 计算机`
   - 支持口语输入：`我数据结构不太行`
   - 支持目标输入：`两周过数据结构`
   - 支持偏好输入：`不要视频，多给我题`

2. 建立画像更新规则
   - 新信息补充画像。
   - 用户纠正覆盖旧信息。
   - 低价值信息只做补充背景。
   - 用户明确输入优先于 mock 和模型推断。

3. 整理智能体协同流程
   - 明确每个 agent 输入什么、输出什么。
   - 明确 agent 之间通过什么字段传递信息。
   - 输出一份流程说明或流程图。

4. 设计 2-3 个核心智能体闭环
   - 画像更新智能体。
   - 知识诊断智能体。
   - 路径资源智能体。
   - 具体接口封装可由后端成员一起实现。
   - 其他智能体先保留设计，不急着全部做实。

5. 建立对话测试集
   - 至少 30 条测试语句。
   - 覆盖画像补充、课程纠正、资源偏好、时间变化、无意义闲聊。
   - 每次改完后跑测试，不靠人工一句句试。

6. 检查智能体输出质量
   - 是否混入错误课程。
   - 是否重复使用 mock 数据。
   - 是否能根据当前课程生成路径和资源。

### 5.2 不需要组长做什么

组长不负责：

- 页面样式细节。
- 数据库表的全部实现。
- 所有 API 的增删改查。
- 前端加载状态和页面布局。
- 后端智能体接口封装的全部实现。

这些交给对应成员，不然任务会堆到组长一个人身上。

### 5.3 组长交付物

- `ProfileAgent` 或画像抽取逻辑升级。
- 核心智能体闭环说明。
- 多轮对话测试集。
- Agent 协同流程说明。
- 智能体输出质量检查结果。

## 6. 后端任务：智能体接口化与数据链路

### 6.1 要做什么

后端负责“前端能不能通过稳定接口调用智能体、数据是不是从壳子变成真实可追踪的数据”。

具体任务：

1. 智能体接口化
   - 封装 `AgentOrchestrator` 调用。
   - 让前端可以通过 API 触发画像更新、路径生成、资源生成。
   - 保证接口返回结构稳定。
   - 处理智能体运行中的错误和超时。

2. 完善 SQLite 基础持久化
   - 保存 session。
   - 保存聊天记录。
   - 保存学生画像快照。
   - 保存学习路径。
   - 保存资源库。
   - 保存学习事件。

3. 建立数据真实性链路
   - 聊天后写入数据库。
   - 画像更新后写入数据库。
   - 路径生成后写入数据库。
   - 资源生成后写入数据库。
   - 页面查询时从数据库或当前 session 最新结果读取。

4. 保证 session 隔离
   - 所有接口都必须接收 `sessionId`。
   - 不同 session 的画像、路径、资源不能混。
   - 不能默认全部落到 `frontend_session_001`。

5. 完善核心 API
   - `GET /api/profile?sessionId=xxx`
   - `GET /api/learning-path?sessionId=xxx`
   - `GET /api/resources?sessionId=xxx`
   - `POST /api/chat/stream`
   - `POST /api/feedback/event`
   - `POST /api/agents/run`

6. 支持学习行为追踪
   - 打开资源。
   - 收藏资源。
   - 完成路径节点。
   - 提交题目结果。
   - 反馈太难、太简单、不想看视频等。

7. 提供后端运行说明
   - 如何创建虚拟环境。
   - 如何安装依赖。
   - 如何启动后端。
   - 如果遇到 404 或数据库错误怎么排查。

### 6.2 后端需要重点注意

- 数据库文件不要提交到 Git。
- `requirements.txt` 必须完整。
- 数据库表初始化不能依赖某一种启动方式。
- API 返回格式要稳定，前端不能每次猜字段。
- mock 数据只能兜底，不能覆盖数据库中的真实数据。
- 智能体接口和普通数据接口要区分清楚：一个负责触发生成，一个负责读取结果。

### 6.3 后端交付物

- 智能体接口可用。
- 基础数据库持久化可用。
- 数据真实更新链路说明。
- 核心 API 可用。
- 后端运行说明。
- 简单接口测试截图或测试说明。

## 7. 前端任务：交互分析与页面同步

### 7.1 要做什么

前端负责“用户怎么看懂系统、怎么知道数据是真的更新了、怎么继续操作”。

具体任务：

1. 分析学习交互流程
   - 用户第一次进入该看到什么。
   - 用户画像不完整时页面怎么提示。
   - 用户生成方案后怎么查看路径和资源。
   - 用户继续补充信息后页面怎么体现变化。

2. 完善页面同步
   - 聊天生成后画像页自动刷新。
   - 学习路径页自动刷新。
   - 资源库页自动刷新。
   - 切换页面时不展示旧数据。
   - 新数据更新时给出明显但不打扰的提示。

3. 增加数据来源提示
   - 用户提供。
   - 智能体生成。
   - 系统推断。
   - 示例/mock 数据。

4. 增加数据更新时间提示
   - 画像更新时间。
   - 路径更新时间。
   - 资源生成时间。
   - 最近一次对话触发了哪些数据更新。

5. 优化画像页
   - 展示画像完整度。
   - 展示缺失字段。
   - 展示最近更新时间。
   - 提供“继续补充画像”的入口。

6. 优化资源库
   - 显示资源类型。
   - 显示关联知识点。
   - 显示资源生成时间。
   - 支持筛选真实生成资源和示例资源。
   - 支持区分教材/讲义、练习题、代码实验、思维导图、视频链接/视频脚本。

7. 优化学习路径页
   - 显示路径是否由当前画像生成。
   - 显示每个阶段对应资源。
   - 显示进度和更新时间。

8. 完善资源详情体验
   - 资源卡片点击后能查看完整内容。
   - Markdown、Mermaid、代码块展示要清楚。
   - 题目类资源要能区分题干、选项、答案或解析。
   - 视频脚本/分镜资源要标注“脚本/待生成多模态”。

9. 增加演示友好的交互状态
   - 智能体运行中显示进度。
   - 生成完成后提示可查看画像、路径、资源。
   - 数据为空时引导用户回到对话页补充信息。
   - 接口失败时显示可理解的错误提示，而不是空白。

### 7.2 前端需要重点注意

- 不要只做静态页面。
- 页面要体现“数据更新了”。
- 要能看出哪些内容来自智能体。
- 要能看出哪些内容还是 mock 或示例。
- 加载中、空状态、错误状态都要处理。

### 7.3 前端交付物

- 第二阶段交互流程图或页面说明。
- 画像页更新状态优化。
- 资源库来源标注。
- 学习路径页动态状态展示。
- 数据更新时间和来源标识。
- 资源详情展示。
- 智能体运行进度和演示提示。

## 8. 第二阶段模块关系

建议系统关系如下：

```text
用户输入
  ↓
前端 ChatPage
  ↓
后端 /api/chat/stream
  ↓
IntentAgent 判断意图
  ↓
ProfileAgent 更新画像
  ↓
KnowledgeAgent 匹配课程知识库
  ↓
DiagnosisAgent 诊断薄弱点
  ↓
PlannerAgent 调整学习路径
  ↓
ResourceAgent 增量生成资源
  ↓
ReviewAgent 审核内容
  ↓
数据库保存最新结果
  ↓
前端画像页 / 路径页 / 资源库刷新
```

这里的重点是：

- 智能体负责“生成和判断”。
- 后端负责“智能体接口化、保存和数据真实性”。
- 前端负责“展示、交互和可信状态说明”。

三者缺一不可。

## 9. 第二阶段验收场景

### 场景一：多轮画像更新

输入：


## 10. 第二阶段建议时间安排

第二阶段最多 3 天完成，但不是减少工作量，而是把原本 4-5 天的内容压缩成 3 天高强度并行推进。

三天内仍然要覆盖：

- 多轮画像持续更新。
- 数据库和 API 稳定。
- 页面同步刷新。
- 资源库根据新需求更新。
- 学习行为事件追踪。
- agent 协同关系梳理。
- 数据来源展示和交互优化。
- 基础测试与演示流程整理。

### 三天结束时必须能演示

1. 用户第一次输入画像，系统生成画像、路径、资源。
2. 用户第二次补充薄弱点，画像能更新。
3. 用户第三次修改偏好，资源库能变化。
4. 用户纠正课程，路径和资源不再展示旧课程。
5. 刷新页面后，主要数据不丢。

三天结束时不是“只做核心”，而是要形成可以演示的第二阶段闭环：

- 多轮对话能更新画像。
- 数据库能保存主要数据。
- 页面能看到最新画像、路径、资源。
- 用户纠正课程后旧数据不再污染。
- 用户反馈偏好后资源推荐能变化。
- 有基础测试和演示案例支撑。
- 能说明哪些数据来自数据库、哪些来自智能体生成、哪些仍是 mock 兜底。

## 12. 需要统一的注意事项

1. 不要各自改各自的接口字段。
2. 所有接口都必须带 `sessionId`。
3. 页面展示的数据必须能说明来源。
4. mock 数据只能兜底，不能覆盖用户真实输入。
5. 第二阶段每个人都要提交可验收的功能，不只是文字说明。
6. 每次 push 前至少跑一次自己负责部分的基本测试。
